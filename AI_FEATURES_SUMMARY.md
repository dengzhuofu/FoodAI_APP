# AI 功能实现总结与面试指南

本文档总结了本项目中最新实现的 AI 增强功能，包括 **RAG (检索增强生成)** 和 **LangGraph 智能 Agent**。旨在帮助开发者梳理技术细节，并为潜在的技术面试做准备。

---

## 1. 功能模块概览

我们对后端的 AI 能力进行了深度升级，主要包含以下三个核心模块：

### A. 语义食谱搜索 (Semantic Recipe Search)
*   **痛点解决**: 传统的关键词搜索（如 SQL `LIKE %keyword%`）无法理解用户意图（例如搜“适合感冒吃的”可能搜不到“白粥”）。
*   **解决方案**: 引入向量搜索，将用户查询和食谱内容映射到高维向量空间，计算语义相似度。
*   **效果**: 支持自然语言模糊查询，能根据食材、口味、场景推荐食谱。

### B. 烹饪知识问答 (RAG Knowledge Base)
*   **痛点解决**: 通用大模型在特定领域（如具体的烹饪技巧、食材替代）可能产生幻觉或回答过于笼统。
*   **解决方案**: 构建本地 `cooking_knowledge.json` 知识库，利用 RAG 技术在回答前检索相关知识。
*   **效果**: Agent 能准确回答“切洋葱怎么不流泪”、“没有发酵粉用什么代替”等专业问题。

### C. 智能活动策划 Agent (Event Planner with LangGraph)
*   **痛点解决**: 单轮对话的 Agent 难以处理复杂的长程任务（如“策划一次聚餐”），需要多步推理和工具协调。
*   **解决方案**: 使用 **LangGraph** 构建状态机工作流，将任务拆解为“搜索食谱 -> 检查库存 -> 生成清单 -> 制定日程”四个步骤。
*   **效果**: 用户只需一句指令，系统自动完成全流程规划，输出结构化的活动方案。

## 2. 技术栈与实现思路

### 核心技术栈
*   **Embedding 模型**: `sentence-transformers/all-MiniLM-L6-v2` (轻量级、高性能，生成 384 维向量)。
*   **向量数据库**: `FAISS (Facebook AI Similarity Search)` (本地高效索引，适合中小型数据集)。
*   **Agent 编排**: `LangGraph` (构建有状态的、循环的图结构 Agent)、`LangChain` (工具调用与模型交互)。
*   **后端框架**: `FastAPI` (异步接口), `Tortoise ORM` (数据库操作)。

### 架构设计思路

#### 1. 向量服务层 (`VectorService`)
*   **设计模式**: 单例服务模式。
*   **职责**: 封装了模型加载、索引管理（加载/保存）、文档添加和相似度搜索逻辑。
*   **多源索引**: 通过 Metadata 中的 `type` 字段（`recipe` vs `knowledge`）区分不同类型的数据，支持混合检索或过滤检索。

#### 2. 图工作流设计 (`PlannerGraph`)
我们定义了一个 `StateGraph`，状态 (`PlannerState`) 在各节点间流转：
*   **State**: 包含 `user_request` (输入), `selected_recipes` (中间产物), `inventory` (外部数据), `shopping_list` (输出), `schedule` (输出)。
*   **Nodes (节点)**:
    1.  `plan_menu`: 调用 `VectorService` 进行 RAG 检索，选定食谱。
    2.  `check_inventory`: 查询数据库获取冰箱库存。
    3.  `generate_shopping_list`: 纯逻辑处理，计算差集（食谱所需 - 库存 = 购物清单）。
    4.  `create_schedule`: 格式化输出。
*   **Edges (边)**: 线性流转 (Start -> Menu -> Inventory -> List -> Schedule -> End)。

## 3. 面试官视角：模拟问答 (Q&A)

### Q1: 为什么选择 FAISS 而不是用 pgvector 或 Pinecone？
**参考回答**:
> “在这个阶段，我们的数据量（食谱+知识）在万级以内，`FAISS` 作为本地库非常轻量且无需额外部署服务，查询延迟极低（毫秒级），非常适合快速迭代和验证。如果未来数据量扩展到百万级，或者需要分布式部署，我会考虑迁移到 `Milvus` 或云端 `Pinecone`。”

### Q2: 你的 RAG 系统是如何处理“冷启动”问题的？（即刚部署时没有索引）
**参考回答**:
> “我们在后端设计了一个初始化机制。`VectorService` 在启动时会检查本地是否存在索引文件。如果不存在，它会加载预设的 JSON 数据（食谱和知识库）进行首次嵌入和索引构建。同时，我们提供了一个 `/sync-index` 接口，允许管理员手动触发全量数据的重新索引。”

### Q3: LangGraph 相比于传统的 LangChain Chain 有什么优势？
**参考回答**:
> “传统的 `SequentialChain` 是线性的，难以处理循环或条件跳转。而在‘聚餐策划’这个场景中，未来我们可能需要增加‘审核’步骤：如果用户不喜欢生成的菜单，流程需要回退到‘菜单规划’节点重试。`LangGraph` 基于图论和状态机，天然支持这种**循环（Cyclic）**和**条件分支**，让 Agent 的行为逻辑更像一个真实的工作流，而不是死板的脚本。”

### Q4: 在计算购物清单时，如何处理食材名称不一致的问题？（比如食谱写‘西红柿’，冰箱里是‘番茄’）
**参考回答**:
> “目前版本我们使用了简单的字符串匹配。为了解决您提到的同义词问题，我有两个优化方案：
> 1. **规则映射**: 维护一个同义词词典。
> 2. **语义匹配**: 利用现有的 Embedding 模型，计算食谱食材和库存食材的向量相似度，如果相似度高于阈值（如 0.85）则视为已有，这能很好地解决‘西红柿/番茄’或‘土豆/马铃薯’的问题。”

### Q5: 引入这些 AI 功能后，如何保证接口的响应速度？
**参考回答**:
> “Embedding 计算和 LLM 推理确实耗时。
> 1. 对于**搜索**，FAISS 的检索非常快，主要耗时在将查询词向量化，`all-MiniLM-L6-v2` 模型很小，CPU 上也能在几十毫秒内完成。
> 2. 对于**复杂策划 (Plan Event)**，这是一个长耗时任务。我们在前端设计上可以采用**异步轮询**或 **WebSocket** 推送，或者展示‘正在思考中...’的加载动画，优化用户体验，而不是阻塞 HTTP 请求。”
